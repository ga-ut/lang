파서 진행 전 결정 필요 사항: 리스트 표현

배경
- Gaut에는 선택 타입/nullable이 없고, 현재 AST의 연결 리스트 타입은 빈 리스트를 안전하게 구성하기 어렵다.
- 파서/타입체커/코드젠이 모두 리스트를 필요로 하므로, 리스트 표현을 먼저 확정해야 한다.

옵션 A: 리스트 전용 런타임 타입 도입
- 내용: List<T>를 런타임/타입체커/코드젠에 추가하고, AST는 List 기반으로 유지
- 장점: AST 구조를 Rust와 동일하게 유지
- 단점: 언어/런타임/FFI 확장 범위가 큼

옵션 B: 인덱스 기반 테이블로 AST 재정의
- 내용: 리스트는 i32 인덱스(0=empty)로 표현하고, AST 노드는 별도 테이블에 저장
- 장점: 언어 확장 없이 구현 가능
- 단점: 파서/타입체커/코드젠 전반에 인덱스 테이블 관리 로직 필요

옵션 C: 파서가 바로 C 생성(무AST 파이프라인)
- 내용: 파서가 토큰 스트림에서 직접 C를 생성하고 AST는 최소화
- 장점: 리스트 표현 문제를 회피
- 단점: 타입체커/최적화 확장에 불리

권장
- Option B가 가장 현실적인 경로로 보이나, 방향 합의 필요.
